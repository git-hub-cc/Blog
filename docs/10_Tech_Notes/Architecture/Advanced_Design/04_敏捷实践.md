# 敏捷实践

本文汇总敏捷开发的核心框架与实践方法，包含：敏捷开发、XP极限编程、看板方法、结对编程。

---

## 一、敏捷开发（Agile Development）

### 摘要
敏捷开发并非单一的方法论，而是涵盖多种框架、实践和原则的综合性理念集合。其核心是《敏捷宣言》中提出的价值观和十二条原则，通过迭代、增量的方式实现快速、灵活、高质量的价值交付。

### 按抽象层次分类

#### 1. 哲学与价值观
- **核心来源**：《敏捷宣言》（Agile Manifesto）
- **四大价值观**：
  - **个体和互动** 高于 流程和工具
  - **可工作的软件** 高于 详尽的文档
  - **客户合作** 高于 合同谈判
  - **响应变化** 高于 遵循计划

#### 2. 原则（Principles）
连接"为什么做（价值观）"和"怎么做（框架）"的桥梁，包括尽早并持续交付、欢迎需求变化、频繁交付可工作软件等十二条原则。

#### 3. 框架与方法论
- **Scrum**：以固定时长的迭代（Sprint）为核心，用于管理复杂产品开发
- **Kanban**：强调可视化工作流、限制在制品（WIP）和持续流动
- **XP（极限编程）**：关注软件工程质量和技术卓越
- **SAFe**：用于大型企业中的规模化敏捷框架

#### 4. 实践与技术（工具箱）
用户故事、TDD、CI/CD、结对编程、每日站会、回顾会议等

### 按应用范围与焦点分类

#### 流程管理型
- **Scrum**：通过时间盒和迭代管理复杂性，确保节奏和可预测性
- **Kanban**：可视化工作流，限制WIP，优化价值流动效率
- **Scrum-ban**：结合Scrum结构与Kanban流动性管理

#### 工程技术型
- **XP（极限编程）**：严格工程实践，确保代码高质量、低缺陷
- **FDD**：围绕"特性"组织开发，强调领域建模和短期迭代

#### 规模化敏捷框架
- **SAFe**：大型企业中协调多个敏捷团队，实现业务战略与开发执行对齐
- **LeSS**：用更少规则去规模化Scrum，强调一个产品、一个产品待办列表
- **Nexus**：为3-9个Scrum团队协作开发单一产品设计的扩展
- **Spotify模型**：通过部落、小队、分会、协会促进跨职能自治和知识共享

### 按方法论"重量"分类

| 类型   | 代表              | 特点                                 | 适用场景                 |
| ------ | ----------------- | ------------------------------------ | ------------------------ |
| 轻量级 | Scrum, Kanban, XP | 规则少，灵活性高，易上手，强调自组织 | 小中型团队，创新型项目   |
| 重量级 | SAFe, RUP         | 规则多，结构化强，学习曲线陡峭       | 大型企业，受严格监管行业 |

### 结论
1. **敏捷不是银弹**：没有一种方法能适用于所有情况
2. **混合是常态**：最成功的实施往往是混合模式（如Scrum框架 + XP工程实践 + Kanban可视化）
3. **回归本源**：无论选择何种方法，都不能忘记《敏捷宣言》的价值观和十二条原则

## 二、XP极限编程（Extreme Programming）

### 摘要
XP是一种高度纪律化的敏捷软件开发方法论，由肯特·贝克（Kent Beck）提出，核心目标是通过缩短开发周期、加强协作和频繁交付高质量软件来应对快速变化的需求。

### 五大核心价值观
1. **沟通（Communication）**：持续、坦诚、面对面的沟通，通过结对编程、现场客户等实践体现
2. **简洁（Simplicity）**：只做当前必要的工作，避免过度设计（YAGNI原则）
3. **反馈（Feedback）**：
   - 微观（秒/分钟）：TDD提供即时反馈
   - 中观（小时/天）：CI提供集成反馈，结对提供实时反馈
   - 宏观（周/月）：客户评审和小型发布
4. **勇气（Courage）**：敢于重构、敢于放弃无价值代码、敢于说实话
5. **尊重（Respect）**：团队成员、客户间互相尊重，是高效协作的基础

### 核心实践

#### 团队协作与流程实践
- **计划游戏**：客户定义价值和优先级，开发者评估实现成本
- **小型发布**：尽快交付有价值的可工作软件小版本
- **系统隐喻**：用简单比喻描述系统运作，建立共同词汇
- **可持续节奏**：每周工作时间稳定，避免长期加班
- **现场客户**：客户代表全程参与，随时解答疑问

#### 开发与技术实践
- **TDD**：红-绿-重构循环，先写失败测试，再写代码，再重构
- **结对编程**：两人共同工作，驾驶员编码，导航员审查
- **重构**：持续改进内部代码结构，消除技术债务
- **持续集成（CI）**：频繁集成代码到主干，自动构建和测试

#### 编码与设计实践
- **代码集体所有权**：任何成员可随时改进任何代码
- **编码规范**：统一的编码风格
- **简单设计**：始终选择最简单的设计，只在必要时增加复杂性

### 优缺点

| 优点                              | 缺点                               |
| --------------------------------- | ---------------------------------- |
| 高质量软件（TDD、CI、重构保障）   | 实施难度大，需要高度纪律性         |
| 高客户满意度（现场客户+小型发布） | 对客户深度参与要求高               |
| 高适应性（灵活响应变更）          | 不适用于大型分布式团队             |
| 提升团队生产力和士气              | "简单设计"可能被误用，缺乏前瞻架构 |

## 三、看板方法（Kanban Method）

### 引言
看板方法是用于管理知识工作、优化服务交付流程的演进式改进方法。其根本目标是通过**可视化工作**、**限制在制品（WIP）**、**管理流动**和**持续改进**，提升工作的可预测性、效率和质量。

### 按应用范围和层级分类

| 层级                | 目标                       | 特征                                         |
| ------------------- | -------------------------- | -------------------------------------------- |
| **个人看板**        | 管理个人任务，提高生产力   | 结构简单（待办/进行中/完成），灵活自定义     |
| **团队级看板**      | 协调团队工作，识别瓶颈     | 严格执行WIP限制，列映射实际工作流程          |
| **项目/产品级看板** | 管理端到端交付流程         | 泳道区分工作类型，关注概念到交付的整个价值流 |
| **投资组合级看板**  | 管理多项目/产品组合        | 处理更高级别的举措，连接战略与执行           |
| **企业/组织级看板** | 全企业推广，跨部门流程优化 | 连接多层看板，形成端到端价值流动网络         |

### 按物理形态分类

**物理看板**：白板、便签纸。优点：高可见性、强互动性、低门槛；缺点：地域限制、数据追踪困难。

**数字看板**（Jira, Trello等）：优点：支持远程协作、自动化度量、集成能力强；缺点：不如物理看板直观，工具复杂性。

### 按成熟度分类（KMM）

1. **任务可视化（L0-1）**：仅为了"看到"工作，无WIP限制，作用有限
2. **流程导向（L2）**：引入并严格执行WIP限制，开始关注流动，识别瓶颈
3. **服务交付（L3）**：系统性使用周期时间、前置时间、吞吐率等指标
4. **进化型组织（L4+）**：看板作为组织持续改进和进化的引擎，内化为Kaizen文化

## 四、结对编程（Pair Programming）

### 概述
两名开发人员在同一台计算机上共同完成编程任务。**驾驶员（Driver）**负责编码，**导航员（Navigator）**负责观察、审查和思考方向，两者角色频繁互换。

### 按经验水平组合分类

| 组合          | 特点                                           | 适用场景                   |
| ------------- | ---------------------------------------------- | -------------------------- |
| **专家-专家** | 高效产出，深入探讨，可能产生"设计僵局"         | 核心模块攻坚，关键架构设计 |
| **专家-新手** | 知识传递，教学相长，确保代码规范               | 新人入职培训，技能提升     |
| **新手-新手** | 共同探索，无压力学习，可能对深层缺陷"双双失明" | 较低难度任务，编码练习     |

### 按协作模式/风格分类

1. **驾驶员-导航员**：最经典模式，频繁轮换角色（每15-30分钟）
2. **乒乓模式**：与TDD紧密结合，A写失败测试→B写代码通过→B重构→B写下一个测试→A接手
3. **强力风格结对**：专家（导航员）绝不碰键盘，强制清晰表达思路；适合知识传授
4. **漫游/探索式**：松散自由，无严格角色划分；适合调试复杂Bug、研究陌生系统

### 按工作空间分类
- **同地结对**：高带宽沟通，沉浸感强；受物理空间限制
- **远程结对**（VS Code Live Share, Tuple等）：地理灵活，每人使用自己配置；依赖工具网络

### 按目标与意图分类
- **任务驱动型**：以最高效完成具体任务为目标
- **知识传授型**：主要目的是传递知识技能
- **问题攻坚型**：解决特定复杂技术难题
- **代码审查型**：将结对作为实时代码审查，缩短反馈周期

### 总结
成功的关键在于**情景化应用**：根据待解决的问题、参与者经验、团队目标和物理条件，有意识地选择最合适的结对类型，从而提升代码质量、促进知识共享、加速团队成长。
