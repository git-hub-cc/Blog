### 1. ⚔️ 矛盾点 (Contradictions) 检查表

*目标：找出不同文档、不同角色产出物之间“互相打架”的硬性冲突。*

#### A. 需求与设计的冲突 (PRD vs. UI/UX)

* **[ ] 功能数量对齐：** UI 设计稿中是否出现了 PRD 中未提及的功能按钮（如 PRD 没写“导出”功能，但设计图上有）？反之亦然。
* **[ ] 状态描述一致性：** PRD 中描述的业务状态机（如“待审核”、“已发布”、“已下线”）是否在 UI 交互流程图中都有对应的页面或组件呈现？
* **[ ] 字段一致性：** 需求文档中要求收集的用户信息表单字段，是否与设计稿上的输入框数量和类型完全一致？

#### B. 前后端契约的冲突 (Frontend vs. Backend API)

* **[ ] 数据结构不匹配：** 后端 API 文档定义的数据格式（如 JSON 中的字段名 `user_id`，类型 `String`），是否与前端或数据库设计模型（如 `userId`，类型 `Int`）完全一致？
* **[ ] 过滤与分页逻辑冲突：** 列表数据的分页、排序、搜索是由前端在内存中处理，还是后端在数据库层面处理？双方的文档描述是否一致？
* **[ ] 校验规则冲突：** 前端文档中定义的表单校验规则（如密码长度、正则表达式），是否与后端 API 接口的安全校验规则完全一致？

#### C. 技术约束与业务目标的冲突 (Tech Specs vs. Business Goals)

* **[ ] 性能指标冲突：** PRD 要求实现毫秒级的实时通讯（如局域网文件传输或实时聊天），但技术选型方案中是否选择了不支持全双工通信的协议（如仅使用传统的 HTTP 轮询）？
* **[ ] 环境兼容性冲突：** 业务要求支持离线可用，但技术栈选型（如某些强依赖云端渲染的框架）是否与之相悖？

### 2. 🌫️ 模糊点与缺口 (Ambiguities / Gaps) 检查表

*目标：找出文档中“没说清楚”、“漏掉的边缘场景”或“想当然的假设”。*

#### A. 异常与边缘场景缺失 (Unhappy Paths & Edge Cases)

* **[ ] 空状态 (Empty States) 定义：** 当列表没有数据时（如新用户的聊天记录为空、回收站为空），页面应该显示什么？UI 和 PRD 是否有定义？
* **[ ] 极限值与边界条件：** 上传文件的最大体积限制是多少？文本输入框的最大字符数是多少？分页加载达到最后一页时的行为是什么？
* **[ ] 错误处理与降级策略：** 当第三方服务宕机（如调用的 AI 接口超时或 Dify 报错）或网络断开时，系统应该如何给用户友好的提示？是否需要重试机制？
* **[ ] 并发与冲突处理：** 如果两个用户同时对同一个资源（如同一篇笔记）进行编辑和保存，系统如何处理冲突（如锁定机制、最后写入胜出等）？

#### B. 逻辑闭环缺失 (Logic Loop Gaps)

* **[ ] 逆向操作定义：** 有“创建”是否有“删除”或“归档”？有“授权”是否有“取消授权”？操作后数据是物理删除还是逻辑删除（如更新 `status` 为 `TRASHED`）？
* **[ ] 权限与安全边界：** 文档是否明确定义了不同角色（如管理员、普通用户、访客）对特定功能和 API 的访问权限边界？越权访问时如何处理？
* **[ ] 状态流转机制：** 数据状态改变的触发条件是什么？（例如，一个定时任务、用户的主动点击，还是第三方 Webhook 的回调？）

#### C. 非功能性需求缺失 (Non-Functional Requirements)

* **[ ] 数据迁移与兼容：** 新增功能或修改数据库字段时，老数据如何处理？是否有旧版本数据迁移脚本的规划？
* **[ ] 监控与埋点：** 核心业务流程是否需要定义数据埋点（用于统计点击率、留存率）或系统级别的监控指标（如 CPU 使用率、错误率）？
