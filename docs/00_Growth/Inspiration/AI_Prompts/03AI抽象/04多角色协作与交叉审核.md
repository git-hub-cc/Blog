### 支柱四：多角色协作与交叉审核 (Multi-Agent Orchestration) 检查表

这个阶段的核心目标是：**终结“全能却平庸”的单体 AI 模式，将复杂项目拆解为专业视角的接力赛，并用机器审核机器，逼出最高质量的产出。**

#### 1. 角色分离与职责边界 (Role Separation)

* [ ] **视角纯粹性：** 是否为当前任务分配了最合适的单一专业视角？（例如：在规划阶段，绝对禁止“前端工程师”角色插嘴商业逻辑，一切以“产品经理 (PM)”的 PRD 为准）
* [ ] **能力互斥设计：** 各个角色的指责是否清晰且不重叠？（例如：“技术总监”只负责敲定技术栈和架构图，“后端开发”只负责基于架构图输出具体的 RESTful API 和数据库表结构）
* [ ] **动态按需唤醒：** 是否明确了何时该呼叫哪个角色？（例如：遇到性能瓶颈时，唤醒“架构师”；遇到界面不协调时，唤醒“UI 设计师”）

#### 2. 标准化输入输出流水线 (Standardized I/O Pipeline)

* [ ] **文档即接口 (SOP)：** 上一个角色的输出是否能完美作为下一个角色的输入？（例如：UX 设计师生成的“Markdown 线框图”必须包含足够的信息结构，以便直接喂给 UI 设计师生成“视觉风格指南”）
* [ ] **格式强制约束：** 产出物是否被强制限定了结构？（例如：要求后端工程师输出的 API 接口设计必须严格包含：功能描述、HTTP 方法、URL 路径、请求参数、成功响应 JSON 示例）
* [ ] **无损上下文传递：** 在角色交接时，是否保留了核心上下文而剔除了干扰信息？（例如：给“前端工程师”派发任务时，只提供 UI 规范和对应的后端 API 文档，不需要提供市场调研报告）

#### 3. 交叉审核与冲突解决 (Cross-Review & Audit)

* [ ] **引入对立视角 (QA / 审核者)：** 是否专门设立了“找茬”的角色？（例如：设定一个严格的“审核专家”，专门对比《PRD 文档》和《API 设计文档》，寻找功能缺失或逻辑矛盾）
* [ ] **矛盾结构化报告：** 审核结果是否要求按照固定格式输出？（例如：强制输出包含“矛盾点编号”、“涉及文档”、“具体表现”、“潜在影响”和“解决建议”的标准化审查报告）
* [ ] **仲裁与最终决策机制：** 当两个角色（文档）发生冲突时，是否规定了优先级？（例如：技术实现细节发生冲突时，以“技术总监”的蓝图为最终判定标准；需求边界模糊时，打回给“产品经理”角色重新定义）

#### 4. 全局视角与交付整合 (Global Integration)

* [ ] **文件清单映射 (File Manifest)：** 在所有角色讨论完毕、准备开始写代码前，是否有一个角色（如“团队负责人”）负责统筹全局，输出包含明确【创建】或【修改】操作的全局文件清单？
* [ ] **多端对齐验证：** 前端、后端、运维的方案是否能在最终清单中闭环？（例如：前端调用的接口在后端清单中存在，后端需要的环境变量在运维的 CI/CD 部署策略中已配置）
